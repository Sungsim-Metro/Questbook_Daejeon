# 모험가의 수첩 프로젝트 설계서

## 1. 프로젝트 개요

`모험가의 수첩`은 대전 지역 관광지를 사용자의 현재 위치, 선호도, 이전 퀘스트 수행 이력과 연결해 맞춤형 관광 퀘스트를 제공하는 모바일 웹 기반 관광 게임 서비스이다. 사용자는 주변 관광지를 탐색하고, 장소별 퀘스트를 수행하며, 완료 기록을 뱃지와 레벨, 개인 수첩 형태로 축적한다.

이 프로젝트는 프로젝트 과목 및 공모전 제출을 위한 프로토타입과 확장 설계를 함께 다룬다. 이 문서는 목표 기능, 데이터 정책, 시스템 아키텍처, 네트워크 구조, 보안 원칙, 확장 방향을 정의한다. 현재 저장소에 실제로 구현된 범위와 다음 구현 작업은 `MVP_STATUS.md`에서 관리한다.

### 1.1 문제 정의

대전 관광은 장소 정보 자체보다 사용자가 지금 어디에 있고, 어떤 활동을 선호하며, 다음에 어떤 장소로 이동하면 좋은지 연결해주는 경험이 중요하다. 일반 관광 정보 서비스는 장소 목록을 제공하는 데 그치는 경우가 많아 관광 행동을 지속시키는 동기가 약하다.

`모험가의 수첩`은 관광지를 게임형 퀘스트로 재구성해 다음 문제를 해결한다.

- 현재 위치 주변의 관광지를 사용자의 취향과 이력에 맞게 추천한다.
- 관광지 방문을 퀘스트, XP, 뱃지, 수첩 기록으로 연결해 재방문 동기를 만든다.
- 대전의 자연, 과학, 원도심, 지역 상권, 이동 수단 등 지역 특성을 퀘스트 테마로 표현한다.
- 관광객의 기록을 개인화된 콘텐츠로 남겨 이후 추천과 공유에 활용한다.

### 1.2 목표 사용자

주요 사용자는 대전 지역을 방문하는 관광객, 대전 내 체험형 관광을 원하는 시민, 프로젝트 과목 또는 공모전 평가자이다. MVP에서는 실제 대규모 운영보다 기능 흐름과 서비스 가능성을 보여주는 데 집중한다.

### 1.3 운영 규모 기준

설계상 운영 규모는 전체 사용자 1,000명 이하, 동시 접속 50명 이하로 둔다. 이 수치는 실제 초기 사용자 예측치가 아니라 프로젝트 과목 및 공모전 제출을 위해 넉넉하게 설정한 상한 기준이다. 따라서 초기 인프라는 복잡한 분산 시스템보다 단순한 웹 서버, 앱 서버, PostgreSQL, Redis, 외부 API 연동 구조를 우선한다.

### 1.4 운영 범위와 비용

이 프로젝트는 공모전·과목 제출용 프로토타입이므로 운영 비용을 최소로 둔다. 인프라는 클라우드 무료 티어 또는 제공 크레딧 범위에서 운영하고, 꿈돌이 IP는 공익적 사용 신청 수준으로 사용한다(§4.7).

실금전 리워드(지역 상점 할인 쿠폰·기념품·체험권)와 파트너 정산은 MVP 범위가 아니라 향후 확장으로 둔다. MVP는 뱃지, 모험가의 수첩, 꿈돌이 도감 같은 비금전 보상만으로 운영한다. 따라서 리워드 관련 부정 사용 방지와 정산 설계(§5.5, §7.10~§7.14, §8.4~§8.5, §11.5)는 상업적 리워드를 실제 도입하는 확장 단계에서만 적용한다.

## 2. 문서 범위와 구현 현황 관리

이 문서는 앞으로 구현할 목표 시스템의 설계 기준을 정의한다. 현재 저장소의 실제 구현 상태, 파일 구조, 목업 데이터 사용 범위, 미구현 기능, 다음 작업 목록은 `MVP_STATUS.md`에서 별도로 관리한다.

문서를 분리하는 이유는 다음과 같다.

- 목표 설계 문서가 현재 임시 구현 세부사항에 묶이지 않게 한다.
- 코드 변경으로 자주 바뀌는 구현 현황을 별도 문서에서 빠르게 갱신한다.
- 공모전 또는 과목 제출 시 목표 설계와 구현 진행 상황을 구분해 설명할 수 있게 한다.

## 3. 핵심 사용자 흐름

사용자는 네이버·구글 등 소셜 계정으로 가입·로그인하고, 이때 위치정보·개인정보 수집·이용에 동의한 뒤 아래 흐름을 이용한다(§9.6). 가입은 만 14세 이상만 허용한다.

### 3.1 관광지 추천 흐름

1. 사용자가 모바일 웹에서 현재 위치 사용을 허용한다.
2. 서비스는 현재 좌표, 선택한 관광 카테고리, 사용자의 선호도를 확인한다.
3. 한국관광공사 OpenAPI를 호출해 현재 위치 주변 관광지 후보를 조회한다.
4. 조회 결과를 유저 단위 임시 캐시에 저장한다.
5. 사용자의 완료 퀘스트, 뱃지 이력, 선호 테마, 거리, 관광 카테고리를 기준으로 후보를 필터링한다.
6. 추천 가능한 관광지를 지도와 퀘스트 카드로 표시한다.
7. 사용자가 관광 카테고리, 위치 권역, 탐색 조건을 변경하면 기존 임시 캐시를 폐기하고 OpenAPI를 다시 호출한다.

### 3.2 퀘스트 수행 흐름

1. 사용자가 추천 관광지 또는 퀘스트 목록에서 수행할 퀘스트를 선택한다.
2. 서비스는 퀘스트 유형, 인증 방식, 예상 보상 XP, 연결 뱃지를 보여준다.
3. 사용자가 현장에 도착하면 GPS 반경 조건을 먼저 확인한다.
4. 퀘스트 유형에 따라 사진, 시간대, 이동 거리, 소비 인증 같은 추가 조건을 확인한다.
5. 인증이 통과되면 퀘스트 완료 기록을 저장한다.
6. 완료 XP를 사용자 레벨과 카테고리별 뱃지 진행도에 반영한다.
7. 뱃지 조건을 충족하면 뱃지 또는 뱃지 단계를 갱신한다.
8. 뱃지 단계 조건을 충족하면 연결된 꿈돌이 도감 항목을 해금한다.
9. 완료 결과를 모험가의 수첩에 기록한다.

### 3.3 수첩 기록 흐름

1. 사용자가 완료한 퀘스트가 생기면 장소, 시간, 카테고리, 획득 XP, 뱃지 정보를 기록한다.
2. 수첩 화면은 완료한 퀘스트와 획득 뱃지를 타임라인 또는 카드 형태로 표시한다.
3. 향후 확장에서는 사용자가 수첩 카드를 이미지로 저장하거나 SNS에 공유할 수 있게 한다.

## 4. 목표 기능 설계

### 4.1 위치 기반 관광지 추천

추천 기능은 현재 위치와 한국관광공사 OpenAPI 조회 결과를 기반으로 한다. 서버는 사용자의 좌표와 필터 조건을 받아 OpenAPI를 호출하고, 응답받은 관광지 후보를 유저 단위 임시 캐시에 저장한다.

추천 점수는 다음 요소를 합산해 계산한다.

- 현재 위치와 관광지 사이의 거리
- 사용자가 입력한 관광 선호도
- 사용자가 이미 완료한 퀘스트 카테고리
- 아직 획득하지 않은 뱃지 또는 다음 단계 뱃지
- 관광지 카테고리와 퀘스트 템플릿의 매칭 가능성
- 현재 시간대 또는 특정 퀘스트 수행 가능 조건

초기 MVP 확장에서는 복잡한 머신러닝 추천보다 규칙 기반 점수화를 사용한다. 데이터가 충분히 쌓인 뒤에는 사용자 행동 이력을 활용한 추천 고도화를 검토한다.

### 4.2 퀘스트 생성

퀘스트 생성은 안정성과 제출 가능성을 위해 템플릿 기반 생성을 1순위로 둔다.

#### 4.2.1 템플릿 기반 생성

관광지 카테고리별로 사전 제작한 템플릿을 준비한다.

- 음식점 및 카페: 소비 인증, 사진 인증, 지역 상권 방문 퀘스트
- 산책로 및 공원: GPS 방문, 이동 거리, 자연 관찰 기록 퀘스트
- 박물관 및 전시관: 전시 관람, 키워드 기록, 사진 인증 퀘스트
- 전망대 및 야경 명소: 시간대 조건, 사진 인증, 방문 기록 퀘스트
- 교통 및 이동형 장소: 이동 거리, 경로 완주, 다음 장소 연결 퀘스트

서버는 OpenAPI 관광지 정보의 카테고리, 장소명, 좌표, 설명을 템플릿에 삽입해 퀘스트 후보를 만든다. 생성된 후보는 금칙 조건, 수행 가능성, 보상 XP 범위, 인증 방식이 올바른지 점검한 뒤 사용자에게 제공한다.

#### 4.2.2 Gemini 기반 생성

Gemini는 퀘스트 문구 다양화와 신규 퀘스트 후보 생성을 위한 보조 수단으로 사용한다. AI 생성 결과는 그대로 사용자에게 제공하지 않고 다음 검증을 통과해야 한다.

- 관광지 정보와 무관한 내용 제거
- 위험하거나 불법적인 행동을 요구하는 문구 제거
- 사유지 침입, 과도한 소비 유도, 안전 위험 행동 제한
- 카테고리, 난이도, 예상 소요 시간, 인증 방식 검증
- XP 보상이 사전 정책 범위 안에 있는지 확인

### 4.3 퀘스트 재사용

다른 사용자를 위해 생성되었고 검증을 통과한 퀘스트는 내부 퀘스트 자산으로 저장해 재사용한다. 단, 한국관광공사 OpenAPI에서 받은 관광지 원본 데이터 전체를 저장하는 방식이 아니라, 서비스가 생성하고 검증한 퀘스트 정의와 필요한 최소 참조 정보만 저장한다.

재사용 대상 데이터는 다음으로 제한한다.

- 퀘스트 제목과 설명
- 퀘스트 유형
- 인증 방식
- 보상 XP
- 연결 뱃지 카테고리
- 관광지 식별을 위한 최소 참조값
- 생성 및 검증 상태

### 4.4 퀘스트 인증

퀘스트 인증은 유형별 MVP 통과 판정을 먼저 고정하고, 고도화 판정은 이후에 붙인다.

GPS 인증 기준 좌표는 저장본이 아니라 해당 `contentId`로 조회한 라이브 OpenAPI 또는 30분 캐시 데이터에서 얻는다.

- 방문형: 관광지 좌표 기준 10m에서 50m 반경 내 GPS 도달이면 통과한다. 사진은 선택 첨부로 둔다.
- 이동형: 출발지와 도착지, 이동 거리, 경로 일부를 GPS로 확인한다.
- 활동형: 사진 존재와 완료 시각이 세션 내인지 확인한다. 사진 내용 판정은 하지 않는다.
- 소비형: 영수증 또는 간판 사진에서 OCR로 상호명만 추출하고 기대 장소 또는 `PartnerStore.name`과 퍼지 매칭한다. 시간·위치창이 함께 맞으면 통과하며, 금액·카드·승인번호 등 민감 필드는 추출·저장하지 않고 OCR 원문은 매칭 후 폐기한다.
- 테마형: 체크리스트 완료 또는 여러 장소 방문 조합을 확인한다.

OCR 실패 또는 불일치는 자동 거부하지 않고 `needs_review` 보류나 GPS+시간 보조 통과로 분기한다.

위치 인증의 신뢰도를 위해 GPS 정확도(accuracy)가 임계값을 넘거나 직전 위치 대비 이동 속도가 비정상적으로 빠르면(위치 조작 의심) 통과시키지 않고 보류한다. 다만 실금전 리워드가 없는 MVP에서는 부정 완료로 얻는 이득이 뱃지·꿈돌이 같은 비금전 보상에 한정되므로, 본격적인 부정 사용 방지는 상업적 리워드를 도입하는 확장 단계에서 강화한다(§1.4).

### 4.5 뱃지 및 레벨 시스템

뱃지는 퀘스트 카테고리별로 사전 정의한다. 사용자가 퀘스트를 완료하면 해당 카테고리의 뱃지 진행도에 XP가 누적된다. 기준 점수를 충족하면 뱃지를 획득하거나 다음 단계로 업그레이드한다.

예시 뱃지 체계는 다음과 같다.

- 자연 관찰: 초록 탐험가
- 과학 문화: 과학 탐험가
- 원도심 걷기: 대전 워커
- 지역 상권: 빵지순례자
- 이동형: 타슈 라이더
- 야경 기록: 전망 수집가

레벨은 모든 퀘스트 완료 XP를 합산해 계산한다. 레벨은 사용자 전체 활동량을 나타내고, 뱃지는 사용자의 관광 성향을 나타내는 태그로 활용한다.

### 4.6 모험가의 수첩

모험가의 수첩은 사용자의 관광 경험을 축적하는 개인 기록장이다. 완료한 퀘스트, 방문 장소, 획득 뱃지, 이동 거리, 주요 사진 또는 메모를 시간순으로 보여준다.

수첩 기능의 역할은 다음과 같다.

- 사용자가 자신의 관광 기록을 다시 볼 수 있게 한다.
- 뱃지와 레벨 성장의 근거를 보여준다.
- 이후 추천 시스템에서 사용자의 선호도와 활동 이력으로 활용한다.
- 공유 가능한 시스템 생성 이미지 카드로 확장해 서비스 홍보 효과를 만든다.

공유 카드는 닉네임·장소·뱃지·꿈돌이로 구성하고, 원본 사진은 비공개로 본인 수첩에만 보관한다. 원본 사진 공유는 콘텐츠 모더레이션과 함께 향후 확장한다.

### 4.7 꿈돌이 도감

꿈돌이는 성향(뱃지)을 의인화한 수집형 아바타 도감이다. 새 보상 화폐가 아니며, 이미 획득한 뱃지 단계가 해금 조건이다.

테마별 완성 꿈돌이 디자인을 통째로 해금한다. 파츠 조립은 사용하지 않는다. 레벨은 외형에 반영하지 않는다.

해금은 조건 달성 시 자동 처리한다. 사용자는 도감에서 표시 꿈돌이를 선택한다. 기본값은 최근 해금 꿈돌이다.

렌더링은 variant당 미리 제작한 완성 이미지 1장으로 처리한다.

`꿈돌이 도감` 화면은 해금된 항목, 잠긴 항목 실루엣, 해금 조건을 함께 보여준다.

해금 매핑 예시는 다음과 같다.

| 조건(뱃지) | 해금 꿈돌이 |
| --- | --- |
| market Lv.2 | 제빵 꿈돌이 |
| science Lv.1 | 안경 꿈돌이 |
| science Lv.2 | 플라스크 꿈돌이 |
| nature Lv.2 | 숲 탐험 꿈돌이 |
| mobility Lv.1 | 타슈 꿈돌이 |
| nightview Lv.2 | 야경 꿈돌이 |

IP 라이선스 원칙은 다음과 같다.

- 꿈돌이는 대전시 상징물이므로 사용 전 사전승인이 필요하다.
- 테마별 변형 디자인은 디자인안별 승인 대상이다.
- 공익·비상업 사용은 신청 통과 가능성이 높으나 자동 승인은 아니다.
- 상업적 리워드 확장 시 상업 IP 승인과 수수료가 필요하며, 운영·비용 모델과 공공데이터 법규 검토에 연결한다.
- 승인 지연 대비 백업은 공식 디자인 최소변형 또는 자체 마스코트로 둔다.

## 5. 시스템 아키텍처

### 5.1 목표 아키텍처

이 프로젝트의 논리 아키텍처는 모바일 웹/PWA, 웹 서버, 앱 서버, 데이터베이스, 외부 API 연동 계층으로 구성된다. 이 논리 구성은 MVP baseline과 확장 단계에 공통으로 적용된다.

MVP baseline은 VPC 안에서 로드 밸런서, 웹·앱·데이터·NAT 서브넷 분리, 내부 라우팅을 갖춘 네트워크 구조(§5.3)를 기본으로 포함한다. 이는 운영 구조 명확성, 계층 간 보안 격리, 확장 대비를 위한 것이다. 다만 각 계층의 서버는 최소 규모로 시작한다. 즉 웹 서버 1대, 앱 서버 1대, 관리형 PostgreSQL 데이터베이스 1대, Redis 캐시 1대로 운영한다. 따라서 baseline과 확장 단계의 차이는 네트워크 토폴로지나 DB 엔진이 아니라 서버 대수와 Redis 용량·배치이며, 확장 단계에서는 정량 임계(§5.2)에 따라 웹·앱 서버와 Redis를 독립적으로 증설한다.

```text
모바일 웹/PWA
  ├─ 사용자 위치 및 선호도 입력
  ├─ 추천 관광지와 퀘스트 표시
  ├─ 퀘스트 인증 요청
  └─ 수첩 및 뱃지 화면 표시

웹 서버
  ├─ 정적 파일 제공
  ├─ HTTPS 보안 헤더와 압축 설정
  ├─ 사용자 웹, 파트너 웹, 관리자 웹 라우팅
  └─ API 요청을 앱 서버로 프록시

앱 서버
  ├─ 사용자와 선호도 관리
  ├─ 한국관광공사 OpenAPI 호출
  ├─ Redis 기반 유저 단위 임시 캐시 관리
  ├─ 추천 점수 계산
  ├─ 퀘스트 생성 및 검증
  ├─ 퀘스트 완료 처리
  ├─ 뱃지와 레벨 갱신
  ├─ 수첩 기록 조회
  └─ 리워드 발급과 쿠폰 사용 처리

PostgreSQL 데이터베이스
  ├─ 사용자 프로필
  ├─ 선호도
  ├─ 완료 퀘스트
  ├─ 뱃지와 레벨 진행도
  ├─ 수첩 기록
  ├─ 검증 완료 퀘스트 자산
  └─ 파트너 상점과 리워드 사용 내역

외부 서비스
  ├─ 한국관광공사 OpenAPI
  ├─ NAVER Maps API
  ├─ Gemini API
  └─ 소셜 로그인 제공자(네이버·구글 OAuth/OIDC)
```

PWA 구성은 MVP에서 Web App Manifest(설치·standalone·아이콘)와 서비스워커를 포함한다. 서비스워커는 정적 자산 precache와 마지막 추천 결과 runtime cache를 담당한다. 캐시 전략은 정적 자산, 추천 결과, 향후 완료 큐를 계층 분리해 웹 푸시, 백그라운드 동기화, 오프라인 완료 큐잉을 재작성 없이 얹을 수 있게 설계한다.

### 5.2 인프라 설계

제출용 운영 기준은 네이버 클라우드 플랫폼 사용을 전제로 한다.

- 프론트엔드: 정적 웹 파일 또는 PWA 배포
- 웹 서버: 정적 파일 제공, HTTPS 보안 헤더, API 프록시
- 앱 서버: Python 앱 서버
- 데이터베이스: 관리형 PostgreSQL
- 캐시: Redis(OpenAPI 응답 최소 필드, 유저 단위 30분 TTL)
- 외부 API: 한국관광공사 OpenAPI, NAVER Maps API, Gemini API
- 예상 규모: 전체 사용자 1,000명 이하, 동시 접속 50명 이하

운영 구조를 명확히 하기 위해 웹 서버와 앱 서버는 처음부터 역할을 분리한다. 서버 대수는 각각 1대로 시작하며, 아래 정량 임계를 넘으면 웹 서버, 앱 서버, Redis를 독립적으로 증설한다. 네트워크 구조(§5.3) 자체는 baseline부터 확장 단계까지 동일하게 유지한다.

| 신호 | baseline 유지 기준 | 확장 검토 임계 |
| --- | --- | --- |
| 동시 접속 | 50명 이하 | 100명 초과가 지속될 때 |
| API 응답 p95 | 500ms 이하 | 1초 초과가 지속될 때 |
| 앱 서버 CPU 사용률 | 60% 이하 | 70% 초과가 지속될 때 |
| 한국관광공사 OpenAPI 일일 호출량 | 할당량의 50% 이하 | 할당량의 80%에 근접할 때 |
| 앱 서버 대수 | 1대 | 2대 이상 필요 시 같은 Redis 캐시를 공유하고, 세션은 stateless JWT로 유지(§9.5) |

데이터베이스 엔진은 PostgreSQL로 확정한다. PostgreSQL은 `Private Data Subnet` 안의 관리형 데이터베이스로 배치하고, 사용자 데이터, 퀘스트, 보상, 파트너 데이터를 schema 또는 table 구조로 논리 분리한다. Redis는 같은 Data Subnet의 내부 캐시로 배치해 TourAPI 임시 후보와 재생성 가능한 서버 캐시만 보관한다.

### 5.3 클라우드 네트워크 구조

초기 네트워크는 NCP VPC 안에서 외부 공개 영역, 웹 서버 영역, 앱 서버 영역, 데이터 영역을 분리한다. 외부 사용자는 로드 밸런서를 통해서만 서비스에 접근하고, 웹 서버는 앱 서버로만 API 요청을 전달한다. 앱 서버와 데이터베이스 및 캐시는 인터넷에서 직접 접근할 수 없게 둔다.

```text
사용자 모바일 웹/PWA
  │
  ▼
Global DNS
  │
  ▼
CDN 또는 WAF
  │
  ▼
Public Application Load Balancer
  │
  ▼
Private Web Subnet
  ├─ 웹 서버 1대
  └─ 웹 서버 2대째는 확장 시 추가
        │
        ▼
Internal Application Load Balancer 또는 내부 라우팅
        │
        ▼
Private App Subnet
  ├─ Python 앱 서버 1대
  └─ Python 앱 서버 2대째는 확장 시 추가
        │
        ├─ Private Data Subnet
        │    ├─ 관리형 PostgreSQL
        │    └─ Redis 캐시
        │
        └─ NAT Gateway Subnet
             └─ 한국관광공사 OpenAPI, NAVER Maps REST API, Gemini API 호출
```

초기 VPC 서브넷 예시는 다음과 같다.

| 영역 | 역할 | 예시 CIDR | 외부 직접 접근 |
| --- | --- | --- | --- |
| Load Balancer Subnet | HTTPS 진입점 | `10.30.0.0/24` | 허용 |
| Web Subnet | 정적 파일 제공, API 프록시 | `10.30.10.0/24` | 차단 |
| App Subnet | Python 앱 서버 | `10.30.20.0/24` | 차단 |
| Data Subnet | PostgreSQL과 Redis | `10.30.30.0/24` | 차단 |
| NAT Gateway Subnet | 내부 서버의 외부 API 호출 출구 | `10.30.250.0/24` | 제한 |

### 5.4 Subnet과 NAT Gateway 역할

Subnet은 VPC 내부를 역할별로 나누는 네트워크 구역이다. 이 프로젝트에서는 공개 진입점, 애플리케이션 서버, 데이터 저장소, 외부 API 호출 출구를 서로 다른 subnet으로 나누어 보안 규칙을 단순하게 만든다.

- Load Balancer Subnet은 사용자가 접근하는 유일한 공개 진입점이다.
- Web Subnet은 정적 파일 제공, 보안 헤더 설정, `/api` 프록시를 담당하는 웹 서버 영역이다.
- App Subnet은 Python 앱 서버가 위치하는 내부 영역이다.
- Data Subnet은 PostgreSQL 데이터베이스와 Redis 캐시가 위치하는 내부 영역이다.
- NAT Gateway Subnet은 App Subnet의 서버가 외부 API를 호출할 때 사용하는 출구이다.

NAT Gateway는 private subnet의 서버가 인터넷으로 나가는 요청을 할 수 있게 해준다. 예를 들어 앱 서버는 한국관광공사 OpenAPI, NAVER Maps REST API, Gemini API를 호출해야 하지만, 외부 사용자가 앱 서버에 직접 접근하면 안 된다. NAT Gateway를 사용하면 내부 서버의 외부 요청은 가능하게 하면서 외부에서 내부 서버로 직접 들어오는 경로는 만들지 않을 수 있다.

웹 서버와 앱 서버를 분리하면 정적 파일 제공, TLS 관련 설정, 보안 헤더, 압축, API 프록시, rate limit 같은 웹 계층 책임과 추천, 퀘스트, 인증, 리워드 같은 비즈니스 로직 책임을 분리할 수 있다. 약간의 서버 비용과 운영 지점이 늘어나지만, 지역 상권 연계와 파트너 포털이 추가될 때 사용자 웹, 파트너 웹, 내부 API를 같은 앱 서버에 직접 노출하지 않고 웹 계층에서 경로와 접근 정책을 통제할 수 있다.

### 5.5 지역 상권 연계 확장 인프라

이 절은 상업적 리워드를 도입하는 향후 확장 단계의 인프라이며 MVP 범위가 아니다(§1.4). 지역 상권과 연계하는 단계에서는 기존 사용자 서비스 외에 파트너 상점이 쿠폰과 리워드 사용을 확인할 수 있는 관리 화면이 필요하다. 이 기능은 별도 시스템으로 완전히 분리하기보다 같은 VPC 안에서 웹 계층과 앱 계층의 역할을 확장한다.

```text
일반 사용자
  └─ questbook.example.com
       └─ 웹 서버
            └─ 앱 서버: 추천, 퀘스트, 뱃지, 수첩, 리워드 발급

지역 상점 담당자
  └─ partner.questbook.example.com
       └─ 웹 서버
            └─ 앱 서버: 파트너 로그인, 쿠폰 조회, 사용 처리

운영 관리자
  └─ admin.questbook.example.com
       └─ 웹 서버
            └─ 앱 서버: 파트너 승인, 리워드 정책 관리, 정산용 내역 조회
```

상권 연계 단계에서 추가로 고려할 인프라 구성은 다음과 같다.

- 파트너 포털은 같은 Web Subnet에 배포하되 도메인과 라우팅을 분리한다.
- 파트너 API와 관리자 API는 같은 App Subnet에서 서비스하되 권한과 인증 정책을 분리한다.
- 쿠폰 이미지, 상점 로고, 공유 이미지 카드는 Object Storage에 저장하고 CDN으로 제공한다.
- 쿠폰 사용 처리, QR 코드 검증, 중복 사용 방지는 PostgreSQL 트랜잭션으로 처리한다.
- 사용 완료 알림이나 쿠폰 만료 알림이 필요해지면 메시지 큐 또는 작업 큐를 추가한다.
- 파트너 상점에는 사용자 개인정보 전체를 제공하지 않고 쿠폰 코드, 사용 가능 여부, 사용 시각 등 최소 정보만 보여준다.

### 5.6 캐시 배치 원칙

캐시는 영구 데이터베이스가 아니다. 이 프로젝트의 캐시는 브라우저 캐시, Redis 캐시, PostgreSQL을 구분해서 사용한다.

| 위치 | 저장 대상 | 목적 |
| --- | --- | --- |
| 브라우저 캐시 | 정적 파일, 최근 화면 상태, 마지막 추천 결과 | UX 개선과 재표시 |
| Redis 캐시 | OpenAPI 응답에서 추출한 최소 관광지 후보 필드 | 추천 계산과 퀘스트 생성 재사용 |
| PostgreSQL | 사용자, 선호도, 뱃지, 레벨, 수첩, 공용 퀘스트 | 서비스 영구 상태 저장 |

한국관광공사 OpenAPI 응답은 사용자의 기기에도 일시적으로 표시될 수 있지만, 추천 점수 계산과 퀘스트 생성의 기준 데이터는 서버가 관리해야 한다. 클라이언트 데이터는 조작될 수 있고, OpenAPI Key도 서버에서 보호해야 하기 때문이다. 따라서 브라우저 캐시는 UX 보조 용도로만 사용하고, Redis 캐시는 추천 및 생성 로직을 위한 임시 저장소로 사용한다.

초기 MVP의 서버 캐시는 Redis로 확정한다. OpenAPI 캐시는 유저, 위치 권역, 관광 카테고리, 조회 시점을 기준으로 묶고 기본 TTL은 30분으로 둔다. Redis에는 원본 응답 전체가 아니라 추천 계산과 퀘스트 생성에 필요한 최소 관광지 후보 필드만 저장한다.

사용자 세션은 stateless JWT(§9.5)로 처리하므로 서버 측 세션 저장소가 필요 없다. 따라서 앱 서버를 여러 대로 증설해도 세션 고정(sticky session) 없이 어느 앱 서버로 라우팅되어도 인증이 유지된다. Redis에는 세션이 아니라 OpenAPI 후보 필드 같은 재생성 가능한 임시 데이터만 둔다.

### 5.7 데이터베이스 분리 원칙

초기에는 사용자 데이터와 퀘스트 데이터를 물리적으로 다른 데이터베이스 서버에 나누지 않는다. 하나의 PostgreSQL 데이터베이스 안에서 schema 또는 table prefix로 도메인을 나누는 방식을 우선한다.

```text
auth
  - 사용자 소셜 연동 식별(provider, providerUserId)
  - 사용자 계정 매핑

user
  - 사용자 프로필
  - 선호도

quest
  - 퀘스트 템플릿
  - 검증 완료 공용 재사용 퀘스트
  - 사용자별 추천 및 수행 상태

reward
  - 레벨 정책
  - 뱃지 정의
  - 사용자 뱃지 진행도
  - 지역 상권 리워드와 쿠폰

note
  - 모험가의 수첩 기록

partner
  - 지역 상점 정보
  - 파트너 담당자 계정
  - 쿠폰 사용 처리 내역
```

사용자 auth 영역은 소셜 식별 중심이며 사용자 비밀번호는 저장하지 않는다. 파트너 계정 자격증명은 partner 스키마에 유지한다.

사용자가 퀘스트를 완료하면 완료 기록, XP, 레벨, 뱃지 진행도, 수첩 기록이 함께 갱신된다. 이 흐름은 하나의 트랜잭션으로 처리하는 것이 안전하다. 데이터베이스를 처음부터 유저 DB와 퀘스트 DB로 물리 분리하면 cross-database transaction, 백업, migration, 장애 대응이 복잡해진다.

물리적 분리는 다음 조건이 생겼을 때 검토한다.

- 유저 개인정보를 국가 또는 지역별로 분리해야 하는 경우
- 공용 퀘스트 라이브러리의 읽기 트래픽이 PostgreSQL을 압박하는 경우
- 분석 로그와 운영 데이터의 저장 주기와 조회 패턴이 크게 달라지는 경우
- 인증, 퀘스트, 리워드 도메인을 서로 다른 팀이 독립 운영해야 하는 경우

따라서 현재 단계의 결정은 다음과 같다.

- DB 엔진은 PostgreSQL로 확정한다.
- 캐시는 Redis로 확정한다.
- PostgreSQL과 Redis는 private data subnet에 배치한다.
- 초기에는 하나의 관리형 PostgreSQL 데이터베이스를 사용한다.
- 도메인 분리는 schema 또는 table 구조로 처리한다.
- OpenAPI 응답은 영구 DB가 아니라 Redis 캐시에만 임시 저장한다.

## 6. 데이터 흐름과 저장 정책

### 6.1 한국관광공사 OpenAPI 사용 원칙

원본 요구사항: "특히 OpenAPI의 데이터를 받아와서 서버에 저장 또는 처리 후 저장하여 사용하는 방식이 아닌".

확정 취지는 원본 미저장이 약관상 금지가 아니라, 추천·퀘스트 목록 생성 시 매번 OpenAPI를 라이브 조회하기 때문이다. 신선도 유지, 관광 카탈로그 영구 복제 불필요, 원본 요구사항 준수를 위한 정책이다. 따라서 서비스가 생성한 퀘스트가 장소를 식별·인증하는 최소 필드의 영구 저장은 이 정책에 위배되지 않는다.

한국관광공사 OpenAPI는 이 프로젝트의 필수 데이터 원천이다. 관광지 원본 데이터는 서버 DB에 영구 저장하지 않는다. 사용자가 현재 관광 중 불러온 OpenAPI 응답은 유저 단위 임시 캐시로만 활용한다.


OpenAPI 응답 필드는 다음 기준으로 취급한다.

| 구분 | 필드(예) | 취급 |
| --- | --- | --- |
| 영구 저장 금지(원본 콘텐츠) | `overview`(설명 전문), 이미지 URL, `addr1`, `tel`, 원본 JSON | 저장 안 함 |
| 조회용(라이브·30분 캐시) | `mapx`/`mapy`(좌표), 주소, 카테고리코드 | 매 호출 라이브, 임시 캐시 |
| 식별용(퀘스트에 영구) | `contentId`, `title`(표시명) | `placeReference` |

임시 캐시 정책은 다음과 같다.

- 캐시 범위: 특정 사용자, 현재 위치 권역, 관광 카테고리, 조회 시점에 묶인다.
- 캐시 목적: 같은 관광 흐름 안에서 다음 관광지 추천과 퀘스트 후보 생성에 재사용한다. 특히 사용자가 거리·선호로 선별한 관광 목록을 투어 중 바뀌지 않게 30분간 안정적으로 유지하는 것이 주 목적이므로 캐시는 유저 단위로 둔다. 원본 후보를 권역·카테고리 단위로 공유하는 2계층 캐시는 호출 할당량이 문제될 때 도입한다.
- 폐기 조건: 기본 유효 시간이 지나면 폐기한다. 초기 설계 기본값은 30분이다.
- 재조회 조건: 사용자가 지역을 크게 이동하거나, 관광 카테고리를 변경하거나, 명시적으로 새로고침하면 OpenAPI를 다시 호출한다.
- 저장 금지: 관광지 원본 응답 전체를 서비스 DB에 장기 저장하지 않는다.

이 정책은 OpenAPI를 필수로 사용하면서도, 사용자가 방금 조회한 관광지 후보를 다음 추천에 활용할 수 있게 하기 위한 절충안이다.

### 6.2 영구 저장 대상

서비스 DB에 저장하는 데이터는 사용자 서비스 경험을 구성하는 내부 데이터로 제한한다.

- 사용자 프로필
- 사용자 선호도
- 완료한 퀘스트 이력
- 획득 뱃지와 뱃지 단계
- 레벨과 XP 진행도
- 모험가의 수첩 기록
- 생성 후 검증된 퀘스트 정의
- 퀘스트 추천과 완료 처리를 위한 최소 참조 정보
- 지역 상점 파트너 정보
- 리워드와 쿠폰 발급 및 사용 처리 내역

### 6.3 저장하지 않는 대상

다음 데이터는 원칙적으로 영구 저장하지 않는다.

- 한국관광공사 OpenAPI 원본 응답 전체
- 사용자의 실시간 위치 이동 경로 전체
- 사진 인증 원본에 포함된 불필요한 개인정보
- API Key, access token, secret 값
- 인증에 필요하지 않은 영수증 상세 정보

### 6.4 공공데이터 출처표시

한국관광공사 OpenAPI(TourAPI)는 공공누리 공공저작물이므로 출처표시가 의무이다.

- 관광지, 추천, 지도, 상세 등 OpenAPI 데이터가 표시되는 화면에는 화면 단위로 "관광정보 제공: 한국관광공사(TourAPI)" 출처를 표기한다. 퀘스트 카드마다 개별 표기할 필요는 없다.
- SNS 공유 카드는 앱 외부로 배포되므로 카드 자체에 작은 출처 캡션을 포함한다.

## 7. 주요 데이터 모델

아래 모델은 설계 기준이며, 실제 구현 시 데이터베이스 스키마와 API 응답 형식에 맞게 조정한다. 카테고리 분류는 여러 모델이 공유하므로 `Category`(§7.19)를 공통 참조로 둔다.

사용자별 소유 관계(완료 퀘스트, 뱃지, 꿈돌이 등)는 소유 목록을 `User` 레코드에 배열로 내장하지 않고, 각 하위 모델이 `userId` 외래 키를 갖는 정규화 구조로 둔다. 조회는 `userId` 인덱스로 처리하므로 목록을 찾는 비용이 작고(전체 스캔이 아니라 인덱스 탐색), 개별 항목의 유일성·외래 키 제약과 동시성 안전성을 유지할 수 있다. 목록을 `User`에 중복 내장하면 row가 무한히 커지고 정본이 이원화되며 항목별 제약을 강제할 수 없다. 표시용 집계가 필요하면 목록 자체가 아니라 정본에서 파생한 카운터만 캐시한다. 주요 조회 인덱스는 `UserQuestInstance(userId, status)`, `QuestCompletion(userId)`, `UserBadge(userId, badgeDefinitionId)`, `AdventureNote(userId, createdAt)`, `UserGgumdori(userId)`이다.

### 7.1 User

사용자 기본 정보를 나타낸다.

- `id`: 사용자 식별자
- `nickname`: 화면에 표시할 이름
- `createdAt`: 가입 시각
- `lastActiveAt`: 마지막 활동 시각

레벨과 누적 XP 같은 진행도 값은 `User`에 중복 저장하지 않고 `LevelProgress`(§7.8)를 단일 정본으로 사용한다.

### 7.2 Preference

사용자의 관광 선호도를 나타낸다.

- `userId`: 사용자 식별자이며 기본 키로 사용한다(사용자당 1개, 1:1)
- `categories`: 선호 관광 카테고리 코드 목록(`Category` 참조, §7.19)
- `distanceRange`: 추천 거리 범위
- `pace`: 선호 활동 강도
- `updatedAt`: 선호도 갱신 시각

### 7.3 TourPlaceCache

한국관광공사 OpenAPI 응답에서 추출한 관광지 후보와 사용자가 거리·선호로 선별한 관광 목록을 유저 단위로 임시 보관하는 캐시이다. 투어 중 목록이 바뀌지 않도록 30분간 안정적으로 유지한다. 구조와 키는 변경하지 않으며 `userId`를 유지한다.

- `userId`: 사용자 식별자
- `regionKey`: 조회 지역 또는 좌표 권역
- `categoryKey`: 관광 카테고리
- `fetchedAt`: OpenAPI 호출 시각
- `expiresAt`: 캐시 만료 시각
- `places`: 추천 계산에 필요한 관광지 후보 목록

`TourPlaceCache`는 영구 관광지 DB가 아니라 현재 관광 흐름 안에서만 유지되는 임시 데이터이다.

### 7.4 ReusableQuest

검증을 통과해 모든 사용자에게 재사용할 수 있는 공용 퀘스트 자산이다. 특정 사용자의 과거 퀘스트 목록이 아니라, 어떤 사용자 요청 과정에서 생성되었더라도 이후 다른 사용자에게 추천할 수 있는 내부 퀘스트 라이브러리로 관리한다.

- `id`: 공용 퀘스트 식별자
- `title`: 퀘스트 제목
- `description`: 퀘스트 설명
- `type`: 방문형, 이동형, 활동형, 소비형, 테마형 중 하나
- `categoryCode`: 뱃지와 추천에 사용할 카테고리(`Category` 참조, §7.19)
- `rewardXp`: 완료 보상 XP
- `verificationType`: GPS, 사진, 시간, 이동 거리, 소비 인증 등
- `placeReference`: 관광지 최소 참조 정보 `{ contentId, placeName }`. `contentId`는 TourAPI 전국 유일 고유번호로 조인 키이며, `placeName`은 표시·폴백용이다
- `source`: 템플릿, Gemini 중 하나
- `reviewStatus`: 생성 결과 검증 상태
- `createdForUserId`: 최초 생성 요청과 연결된 사용자 식별자
- `isReusable`: 다른 사용자에게 추천 가능한지 여부
- `reuseCount`: 추천 또는 할당에 재사용된 횟수
- `completionCount`: 완료된 횟수

`placeReference`에는 한국관광공사 OpenAPI 원본 응답 전체를 저장하지 않는다. 좌표, 주소, 설명, 이미지는 저장하지 않고 `contentId`로 라이브 조회한다. 이름이 겹치는 관광지가 있어도 `contentId`가 유일하므로 정확히 식별·매칭된다.

### 7.5 UserQuestInstance

공용 퀘스트가 특정 사용자에게 추천되거나 사용자가 수행하기로 선택했을 때 생성되는 사용자별 퀘스트 상태이다.

- `id`: 사용자별 퀘스트 인스턴스 식별자
- `userId`: 사용자 식별자
- `reusableQuestId`: 공용 퀘스트 식별자
- `status`: 추천됨, 수락됨, 진행 중, 완료됨, 포기됨 중 하나
- `recommendedAt`: 추천 시각
- `acceptedAt`: 사용자가 수행하기로 선택한 시각
- `expiresAt`: 추천 또는 수행 가능 만료 시각
- `completedAt`: 완료 시각

이 모델을 분리하면 같은 공용 퀘스트를 여러 사용자가 수행하더라도 각 사용자의 진행 상태와 완료 여부를 독립적으로 관리할 수 있다.

### 7.6 QuestCompletion

사용자가 퀘스트를 완료한 기록이다.

- `id`: 완료 기록 식별자
- `userId`: 사용자 식별자
- `userQuestInstanceId`: 사용자별 퀘스트 인스턴스 식별자
- `reusableQuestId`: 공용 퀘스트 식별자
- `completedAt`: 완료 시각
- `earnedXp`: 획득 XP
- `verificationResult`: 인증 결과
- `photoRef`: 선택 인증 사진 참조. EXIF 제거 후 접근제어 저장소에 보관한다.
- `noteId`: 생성된 수첩 기록 식별자

`verificationResult`에는 `{method, ocrMatchedStore, matchScore, decision}`을 포함할 수 있다.

`reusableQuestId`는 `userQuestInstanceId`를 통해 도출할 수 있는 조회 편의용 역정규화 필드이며, 정본 연결은 `userQuestInstanceId`이다. `noteId`와 `AdventureNote.questCompletionId`는 서로를 참조하므로 완료 처리 트랜잭션(§8.3)에서 함께 일관되게 설정한다.

### 7.7 Badge (BadgeDefinition / UserBadge)

뱃지는 전역 공유 정의(`BadgeDefinition`)와 사용자별 획득 상태(`UserBadge`)로 분리한다. 정의는 모든 사용자에게 공통이고 상태는 사용자마다 다르므로, 단일 출처를 유지하기 위해 나눈다.

`BadgeDefinition` — 뱃지 정의(전역 공유)

- `id`: 뱃지 정의 식별자
- `categoryCode`: 연결 카테고리(`Category` 참조, §7.19)
- `name`: 뱃지 이름
- `tier`: 뱃지 단계
- `requiredXp`: 해당 단계 획득에 필요한 XP

`UserBadge` — 사용자별 획득 상태

- `id`: 사용자 뱃지 식별자
- `userId`: 사용자 식별자
- `badgeDefinitionId`: 뱃지 정의 식별자
- `progressXp`: 사용자별 진행 XP
- `earnedAt`: 획득 시각(미획득이면 비어 있음)

`(userId, badgeDefinitionId)` 조합은 유일하다. 꿈돌이 도감의 해금 조건(§7.15)은 이 `BadgeDefinition.categoryCode`와 `tier`를 기준으로 판정한다.

### 7.8 LevelProgress

사용자 레벨·XP 진행도의 단일 정본이다(§7.1 참조). 레벨과 누적 XP는 이 모델에만 저장한다.

- `userId`: 사용자 식별자이며 유일하다(사용자당 1개)
- `currentLevel`: 현재 레벨
- `currentXp`: 현재 레벨 내 XP
- `totalXp`: 누적 XP
- `nextLevelRequiredXp`: 다음 레벨까지 필요한 XP

### 7.9 AdventureNote

모험가의 수첩에 표시할 사용자 기록이다.

- `id`: 수첩 기록 식별자
- `userId`: 사용자 식별자
- `reusableQuestId`: 연결된 공용 퀘스트 식별자
- `questCompletionId`: 연결된 완료 기록 식별자
- `placeName`: 표시용 장소명
- `summary`: 활동 요약
- `badges`: 연결된 뱃지 목록
- `distanceKm`: 기록된 이동 거리
- `createdAt`: 기록 생성 시각
- `shareImageUrl`: 시스템 생성 공유 이미지 카드 경로. 카드는 닉네임·장소·뱃지·꿈돌이로 구성한다.

원본 사진은 비공개로 본인 수첩에 보관한다. 원본 사진 공유는 콘텐츠 모더레이션과 함께 향후 확장한다.

### 7.10 PartnerStore

지역 상권 연계 단계에서 쿠폰이나 리워드를 제공하는 상점 또는 기관 정보이다. 아래 §7.10~§7.14 모델은 상업적 리워드를 도입하는 향후 확장 단계에서만 사용하며 MVP 범위에는 포함되지 않는다(§1.4).

- `id`: 파트너 상점 식별자
- `name`: 상점 또는 기관 이름
- `category`: 음식점, 카페, 전통시장, 체험 공간, 기념품점 등 분류
- `area`: 대전 내 권역 또는 행정동
- `address`: 표시용 주소
- `contactPhone`: 운영 연락처
- `status`: 신청됨, 승인됨, 중지됨 중 하나
- `createdAt`: 등록 시각
- `updatedAt`: 갱신 시각

파트너 상점 정보는 OpenAPI 관광지 원본을 복제하는 용도가 아니라, 서비스가 직접 제휴한 상점과 리워드 제공 조건을 관리하기 위한 내부 운영 데이터이다.

### 7.11 PartnerAccount

지역 상점 담당자가 파트너 포털에서 쿠폰 사용 여부를 확인하고 사용 처리할 수 있게 하는 계정이다.

- `id`: 파트너 계정 식별자
- `partnerStoreId`: 소속 파트너 상점 식별자
- `loginId`: 로그인 ID
- `passwordHash`: 비밀번호 해시
- `role`: 소유자, 직원, 조회 전용 중 하나
- `status`: 활성, 잠김, 비활성 중 하나
- `lastLoginAt`: 마지막 로그인 시각
- `createdAt`: 생성 시각

파트너 계정도 사용자 계정과 마찬가지로 비밀번호 원문을 저장하지 않는다.

### 7.12 RewardOffer

사용자에게 발급할 수 있는 쿠폰, 할인권, 기념품 교환권, 체험권 같은 리워드 정의이다.

- `id`: 리워드 식별자
- `partnerStoreId`: 제공 파트너 상점 식별자
- `title`: 리워드 제목
- `description`: 리워드 설명
- `rewardType`: 할인 쿠폰, 무료 제공, 기념품, 체험권 등 유형
- `issueCondition`: 발급 조건
- `validFrom`: 사용 가능 시작 시각
- `validUntil`: 사용 가능 종료 시각
- `totalQuantity`: 총 발급 가능 수량
- `issuedQuantity`: 발급된 수량
- `status`: 준비 중, 활성, 소진, 종료, 중지 중 하나

발급 조건은 특정 퀘스트 완료, 특정 뱃지 획득, 특정 카테고리 XP 달성, 기간 한정 이벤트 등으로 정의할 수 있다.

`issuedQuantity`는 `totalQuantity`를 초과할 수 없다. 이 불변식은 CHECK 제약 또는 조건부 UPDATE(발급 시 잔여 수량 확인)로 강제해 동시 발급에서 초과 발급을 막는다.

### 7.13 UserReward

특정 사용자에게 실제로 발급된 리워드 또는 쿠폰이다.

- `id`: 사용자 리워드 식별자
- `userId`: 사용자 식별자
- `rewardOfferId`: 리워드 정의 식별자
- `status`: 발급됨, 사용됨, 만료됨, 취소됨 중 하나
- `issueReason`: 발급 사유
- `issuedAt`: 발급 시각
- `expiresAt`: 만료 시각
- `redeemedAt`: 사용 처리 시각
- `redemptionCodeHash`: 쿠폰 코드 해시

쿠폰 코드는 중복 사용 방지를 위해 서버에서 생성하고, DB에는 원문 대신 검증 가능한 해시 또는 별도 보호된 값을 저장한다. 사용자가 화면에서 제시하는 QR 코드나 숫자 코드는 `RewardRedemption` 처리 시 검증한다.

`redemptionCodeHash`는 유일하다. 동일 사용자에게 같은 `rewardOfferId`가 중복 발급되지 않도록 `(userId, rewardOfferId)` 유일 제약을 두거나, 리워드별 사용자당 발급 한도(perUserLimit)로 제어한다.

### 7.14 RewardRedemption

파트너 상점에서 쿠폰 또는 리워드를 실제로 사용 처리한 기록이다.

- `id`: 사용 처리 기록 식별자
- `userRewardId`: 사용자 리워드 식별자
- `partnerStoreId`: 사용 처리 상점 식별자
- `partnerAccountId`: 사용 처리 담당자 계정 식별자
- `redeemedAt`: 사용 처리 시각
- `status`: 성공, 실패, 취소 중 하나
- `failureReason`: 실패 사유
- `requestMetadata`: 사용 처리 요청의 최소 진단 정보

`userRewardId`는 유일하다(쿠폰 1건당 성공 사용 처리는 1회). 리워드 사용 처리는 중복 사용이 발생하지 않도록 PostgreSQL 트랜잭션 안에서 `UserReward` 상태 변경과 `RewardRedemption` 생성을 함께 처리한다.

### 7.15 GgumdoriVariant

꿈돌이 도감에 표시할 완성형 아바타 변형이다.

- `id`: 꿈돌이 변형 식별자
- `name`: 꿈돌이 이름
- `themeCategory`: 테마 분류. 뱃지 카테고리와 동일한 `Category` 코드를 사용한다(§7.19)
- `tier`: 연결 뱃지 단계
- `unlockCondition`: 해금 조건
- `imageRef`: 완성 이미지 참조
- `description`: 설명 문구
- `rarity`: 표시 희귀도
- `sortOrder`: 도감 정렬 순서

### 7.16 UserGgumdori

사용자가 해금한 꿈돌이 항목이다.

- `id`: 사용자 꿈돌이 식별자
- `userId`: 사용자 식별자
- `variantId`: 꿈돌이 변형 식별자
- `unlockedAt`: 해금 시각

`userId`와 `variantId` 조합은 유일해야 한다.

### 7.17 GgumdoriSelection

사용자가 현재 표시 대상으로 선택한 꿈돌이다.

- `userId`: 사용자 식별자이며 기본 키로 사용한다.
- `selectedVariantId`: 선택된 꿈돌이 변형 식별자
- `updatedAt`: 선택 갱신 시각

### 7.18 UserAccount

사용자의 소셜 로그인 연동 계정 정보이다. 회원가입과 로그인은 네이버·구글 등 소셜 계정 연동(OAuth 2.0 / OIDC)으로 처리하며 자체 비밀번호는 두지 않는다.

- `id`: 계정 식별자
- `userId`: 연결된 `User` 식별자(1:1)
- `provider`: 소셜 로그인 제공자(naver, google 등)
- `providerUserId`: 제공자가 부여한 사용자 식별자
- `email`: 이메일(선택)
- `displayName`: 표시 이름(선택)
- `createdAt`: 생성 시각
- `lastLoginAt`: 마지막 로그인 시각

`(provider, providerUserId)` 조합은 유일하며, 소셜 로그인이므로 비밀번호는 저장하지 않는다.

### 7.19 Category

관광·뱃지·꿈돌이 테마에 공통으로 쓰는 카테고리 분류 참조 모델이다. 여러 모델이 카테고리를 문자열로 각자 보관하면 값이 어긋날 수 있으므로, 고정된 코드 집합을 한 곳에서 정의하고 다른 모델은 코드로 참조한다.

- `code`: 카테고리 코드이며 기본 키로 사용한다(예: `nature`, `science`, `market`, `downtown`, `mobility`, `nightview`)
- `name`: 화면 표시 이름
- `description`: 설명 문구
- `sortOrder`: 표시 정렬 순서

`Preference.categories`, `ReusableQuest.categoryCode`, `BadgeDefinition.categoryCode`, `GgumdoriVariant.themeCategory`는 모두 이 `Category.code`를 참조한다. 이를 통해 퀘스트 카테고리, 뱃지, 꿈돌이 테마가 같은 분류 체계로 일관되게 연결된다.

## 8. API 흐름 설계

아래 모든 API는 인증된 사용자 요청을 전제로 하며, 각 흐름은 토큰에서 도출한 사용자 신원으로 본인 자원만 접근한다. 클라이언트가 보낸 사용자 식별자는 신뢰하지 않는다(§9.5).

### 8.1 추천 API 흐름

1. 클라이언트가 현재 위치, 관광 카테고리, 선호 조건을 앱 서버에 전달한다.
2. 앱 서버는 같은 사용자, 같은 지역, 같은 카테고리의 유효한 `TourPlaceCache`가 있는지 확인한다.
3. 유효한 캐시가 있으면 캐시를 사용한다.
4. 캐시가 없거나 만료되었거나 조건이 바뀌었으면 한국관광공사 OpenAPI를 호출한다.
5. OpenAPI 응답에서 추천 계산에 필요한 필드만 추출해 유저 단위 임시 캐시에 저장한다.
6. 앱 서버는 사용자 이력과 선호도를 반영해 추천 점수를 계산한다.
7. 라이브 OpenAPI 후보를 `placeReference.contentId`로 대조해 해당 관광지에 이미 만들어 둔 `ReusableQuest`가 있으면 재사용하고, 없으면 퀘스트 생성 흐름으로 넘긴다.
8. 추천 관광지와 연결 가능한 퀘스트 후보를 클라이언트에 반환한다.

### 8.2 퀘스트 생성 API 흐름

1. 추천 관광지 후보를 입력으로 받는다.
2. 라이브 OpenAPI 후보의 `contentId`를 `ReusableQuest.placeReference.contentId`와 대조한다. 기존 `ReusableQuest`가 있으면 재사용하고 이후 생성 단계를 건너뛴다.
3. 관광지 카테고리에 맞는 퀘스트 템플릿을 선택한다.
4. 장소명, 위치, 카테고리, 설명을 템플릿에 삽입한다.
5. 필요하면 Gemini를 호출해 문구 다양화 또는 추가 후보를 생성한다.
6. 생성 결과를 검증한다.
7. 검증된 퀘스트를 `ReusableQuest`로 저장해 모든 사용자가 활용 가능한 공용 퀘스트 자산으로 만든다.
8. 해당 퀘스트가 특정 사용자에게 추천되거나 선택되면 `UserQuestInstance`를 생성한다.

### 8.3 퀘스트 완료 API 흐름

1. 클라이언트가 퀘스트 완료 요청과 인증 정보를 제출한다.
2. 앱 서버는 인증 방식에 따라 GPS, 시간, 사진, 이동 거리 조건을 확인한다.
3. 인증이 실패하면 실패 사유와 재시도 가능 여부를 반환한다.
4. 인증이 성공하면 `QuestCompletion`을 생성한다.
5. 사용자 XP와 `LevelProgress`를 갱신한다.
6. 연결 카테고리의 `UserBadge` 진행도를 갱신한다.
7. 필요한 경우 새 뱃지 또는 뱃지 단계를 부여한다.
8. 뱃지 단계 조건을 충족하면 대응하는 `GgumdoriVariant`를 해금해 `UserGgumdori`를 생성한다.
9. 연결된 `ReusableQuest`의 완료 횟수와 재사용 통계를 갱신한다.
10. `AdventureNote`를 생성한다.

### 8.4 리워드 발급 API 흐름

이 리워드 흐름(§8.4~§8.5)은 상업적 리워드를 도입하는 향후 확장 단계의 설계이며 MVP 범위가 아니다(§1.4).

1. 퀘스트 완료, 뱃지 획득, 레벨 상승, 이벤트 참여 같은 리워드 발급 트리거가 발생한다.
2. 앱 서버는 활성 상태의 `RewardOffer` 중 발급 조건을 만족하는 항목을 조회한다.
3. 발급 수량, 유효 기간, 사용자별 중복 발급 제한을 확인한다.
4. 조건을 만족하면 `UserReward`를 생성하고 쿠폰 코드 또는 QR 코드 검증값을 발급한다.
5. `RewardOffer.issuedQuantity`를 갱신한다.
6. 사용자 앱은 발급된 리워드 목록과 만료 시각을 표시한다.

### 8.5 리워드 사용 처리 API 흐름

1. 사용자가 지역 상점에서 앱의 쿠폰 QR 코드 또는 사용 코드를 제시한다.
2. 파트너 담당자는 파트너 포털에서 QR 코드 또는 사용 코드를 스캔하거나 입력한다.
3. 앱 서버는 `PartnerAccount` 권한과 `PartnerStore` 상태를 확인한다.
4. `UserReward`가 해당 상점에서 사용 가능한지, 만료되지 않았는지, 이미 사용되지 않았는지 검증한다.
5. 검증이 통과하면 하나의 트랜잭션 안에서 `UserReward` 상태를 사용됨으로 변경하고 `RewardRedemption`을 생성한다.
6. 검증이 실패하면 실패 사유만 반환하고 사용자 개인정보는 파트너 화면에 노출하지 않는다.
7. 운영 관리자는 관리자 화면에서 파트너별 발급 수량, 사용 수량, 실패 사유, 기간별 사용 내역을 조회한다.

## 9. 보안 및 개인정보 설계

### 9.1 API Key 관리

NAVER Maps REST API Key, 한국관광공사 OpenAPI Key, Gemini API Key는 서버 환경변수 또는 배포 환경의 secret 관리 기능에 저장한다. 브라우저에는 공개가 필요한 Key ID 또는 클라이언트용 설정만 전달한다.

NAVER Geocoding과 Reverse Geocoding은 서버 프록시를 통해 호출한다. 이 구조를 유지해 브라우저에 REST API Key가 노출되지 않도록 한다.

회원가입과 로그인은 네이버·구글 등 소셜 계정 연동(OAuth 2.0 / OIDC)으로 처리한다. 소셜 제공자 인증은 서버에서 처리하고, 제공자의 access token과 refresh token은 서버에서만 최소한으로 보관한다. 사용자에게는 서비스 자체 세션을 발급하며 자체 비밀번호는 저장하지 않는다. 세션 유지, 전송 보안, 요청 단위 인가의 상세 설계는 이후 보안 강화 단계에서 확장한다.

### 9.2 위치 정보 보호

사용자 위치는 추천과 인증에 필요한 순간에만 사용한다. 실시간 이동 경로 전체를 기본 저장하지 않으며, 퀘스트 완료에 필요한 최소 좌표 또는 인증 결과만 저장한다.

### 9.3 사진 및 소비 인증 보호

사진 인증과 영수증 인증은 개인정보 노출 가능성이 있다. 향후 구현 시 다음 원칙을 적용한다.

- 인증에 필요한 최소 파일만 업로드한다.
- 영수증의 결제 수단, 승인번호, 개인 식별 정보는 저장하지 않는다.
- 업로드된 모든 사진은 서버 저장 전 EXIF 위치·기기 메타데이터를 기본 제거한다.
- 제출용 MVP에서는 민감한 인증 데이터 저장 없이 인증 흐름을 우선 시연한다.

### 9.4 파트너 포털과 리워드 보안

지역 상점 파트너는 쿠폰 사용 처리에 필요한 최소 정보만 확인할 수 있어야 한다. 파트너 포털에는 사용자의 전체 프로필, 위치 이력, 퀘스트 상세 이력, 연락처를 노출하지 않는다.

- 파트너 담당자는 `PartnerAccount`로 별도 로그인한다.
- 파트너 계정은 자신이 소속된 `PartnerStore`의 리워드만 조회하고 처리할 수 있다.
- 쿠폰 원문 코드는 DB에 그대로 저장하지 않고 해시 또는 별도 보호된 검증값으로 저장한다.
- 쿠폰 사용 처리는 중복 사용을 막기 위해 PostgreSQL 트랜잭션으로 처리한다.
- 관리자 화면은 파트너 승인, 리워드 정책 관리, 사용 내역 조회 권한을 일반 파트너 계정과 분리한다.
- 리워드 사용 로그에는 쿠폰 검증 결과와 진단 정보만 남기고 불필요한 사용자 개인정보는 남기지 않는다.

### 9.5 사용자 인증·세션 보안

소셜 로그인(§9.1)으로 인증한 뒤 세션과 요청 인가를 다음 원칙으로 처리한다.

- 세션: 로그인 성공 시 서비스가 서명된 stateless JWT를 발급하고 Authorization 헤더로 전달받는다. 앱 서버가 여러 대로 늘어도 세션을 공유할 필요가 없다. 만료된 토큰은 refresh 토큰으로 갱신한다.
- 전송 보안: 모든 통신은 HTTPS로만 처리하고 HSTS를 적용한다. 위치·사진 같은 민감 데이터가 평문으로 노출되지 않게 한다.
- CSRF: 인증 토큰을 쿠키가 아니라 Authorization 헤더로 전달해 교차 사이트 요청 위조 표면을 최소화한다. 쿠키를 사용할 경우 SameSite 설정과 CSRF 토큰을 함께 적용한다.
- 요청 단위 인가: 모든 조회·상태 변경 API는 토큰에서 도출한 사용자 신원으로 본인 자원만 접근하게 한다. 클라이언트가 보낸 `userId` 같은 식별자는 신뢰하지 않는다.
- 입력 검증: 좌표, 사진, 인증 코드 등 외부 입력은 서버에서 형식과 범위를 검증한다.

### 9.6 법규 적합성 및 개인정보 처리

위치와 개인정보를 다루므로 위치정보법과 개인정보보호법을 따른다.

- 동의: 가입 또는 최초 위치 사용 시 수집·이용 목적과 항목, 보관기간을 고지하고 동의를 받는다. 동의를 철회하거나 위치 권한을 비활성화하면 즉시 수집을 중단하고 관련 기능을 제한한다.
- 수집 최소화: 소셜 로그인 식별자, 닉네임, 이메일(선택) 등 서비스에 필요한 최소 항목만 수집한다. 위치는 추천·인증에 필요한 순간에만 사용하고 최소만 저장한다(§9.2).
- 개인정보 처리방침: 수집 항목, 이용 목적, 보관기간, 파기 방법, 제3자 제공 여부를 담은 처리방침을 게시한다.
- 보관·파기: 회원 탈퇴 시 사용자 계정과 연관 데이터(완료 기록, 수첩, 꿈돌이 해금, 발급 리워드 등)를 파기하거나 비식별 처리한다. 법정 보관 의무가 없는 항목은 보관기간 경과 시 파기한다.
- 미성년자: 가입 시 연령을 확인하고 만 14세 미만은 가입을 제한한다.
- 위치기반서비스 신고: 위치기반서비스를 실제 사업으로 운영하려면 방송통신위원회 신고가 필요하다. 본 프로토타입은 공모전·과제 시연 범위이므로, 실운영 전에 신고와 처리방침 게시 등 법적 절차를 완료하는 것을 전제로 한다.

## 10. 운영 및 장애 대응

### 10.1 외부 API 복원력과 장애 대응

외부 API(한국관광공사 OpenAPI, NAVER Maps, Gemini)는 서비스 가용성에 직접 영향을 주므로 호출마다 복원력 정책을 적용한다.

- 타임아웃: API별로 연결·응답 타임아웃 상한을 두어 느린 응답 하나가 전체 요청을 묶지 않게 한다.
- 재시도: 일시적 오류(5xx, 타임아웃)에 한해 지수 백오프로 제한된 횟수만 재시도하고, 4xx 같은 영구 오류는 재시도하지 않는다.
- 서킷 브레이커: 특정 API의 연속 실패율이 임계를 넘으면 일정 시간 호출을 차단하고 곧바로 fallback으로 전환해 장애가 전체로 번지지 않게 한다.
- 호출 쿼터 관리: API별 일일 호출 한도를 추적하고, 유저 단위 임시 캐시(§5.6)로 중복 호출을 줄여 쿼터 소진을 방지한다.

위 정책으로도 호출이 실패하면 다음과 같이 fallback한다.

한국관광공사 OpenAPI 호출이 실패하면 사용자에게 재시도 안내와 기존 화면 유지 상태를 제공한다. 유효한 유저 단위 임시 캐시가 있으면 해당 캐시를 사용하되, 만료된 원본 데이터를 장기 저장소처럼 사용하지 않는다.

NAVER Maps 로드에 실패하면 목업 지도 또는 장소 목록 중심 화면으로 fallback한다.

Gemini 호출이 실패하면 템플릿 기반 퀘스트만 제공한다.

### 10.2 성능 기준

동시 접속 50명 이하를 기준으로 초기 웹 서버와 앱 서버는 각각 단일 인스턴스로 처리 가능하다. 다만 웹 서버와 앱 서버는 역할과 배포 단위를 분리해 두고, 트래픽이 늘어나면 웹 서버와 앱 서버를 독립적으로 증설한다. OpenAPI 호출은 사용자 요청마다 무조건 반복하지 않고 유저 단위 임시 캐시를 사용해 외부 API 호출량을 줄인다.

### 10.3 로그 정책

운영 로그에는 요청 성공 여부, 외부 API 호출 결과, 오류 유형, 응답 시간 등 진단에 필요한 정보만 남긴다. API Key, 원본 인증 파일, 민감한 위치 상세 이력은 로그에 남기지 않는다.

### 10.4 배포와 CI/CD

소스 저장소 push를 기준으로 빌드, 정적 검사, 핵심 테스트, 배포를 자동화한다. baseline에서는 단순 파이프라인(빌드 → 검사·테스트 → 배포)으로 시작한다. 무중단 배포(rolling 또는 blue-green)는 앱 서버가 여러 대로 늘어나는 확장 단계에서 도입한다. 배포 산출물과 환경 설정은 버전 관리하되, API Key 등 비밀 값은 코드·저장소에 포함하지 않고 별도 비밀 관리에서 주입한다(§9.1).

### 10.5 모니터링·헬스체크·알람

웹 서버와 앱 서버는 헬스체크 엔드포인트를 제공해 로드 밸런서가 비정상 인스턴스를 트래픽에서 제외할 수 있게 한다. 요청 성공률, 응답 지연(p95), 외부 API 실패율, 서버 자원 사용률 같은 핵심 지표를 수집하고, 임계를 넘으면 담당자에게 알람을 보낸다. baseline에서는 NCP Cloud Insight 등 관리형 모니터링과 기본 지표로 시작하고, 필요에 따라 상세 대시보드를 확장한다.

### 10.6 백업과 복구

PostgreSQL은 자동 백업을 활성화하고 최소 하루 1회 스냅샷을 보관한다. baseline 목표는 RPO 24시간, RTO 수 시간 수준이며, 정식 운영 단계에서 백업 주기 단축과 복구 리허설로 강화한다. Redis 캐시는 재생성 가능한 임시 데이터만 저장하므로 백업 대상이 아니다. 이 운영 항목들은 정식 서비스 운영 전 이행을 전제로 하며, 프로토타입 시연 범위에서는 설계 수준으로 명시한다.

## 11. 향후 개발 단계

### 11.1 1단계: 현재 MVP 정리

- 정적 화면과 목업 데이터를 유지하며 제출 가능한 흐름을 안정화한다.
- NAVER Maps 설정과 fallback 동작을 검수한다.
- README와 설계 문서를 정리한다.

### 11.2 2단계: OpenAPI 추천 기능 연결

- 웹 서버와 앱 서버를 분리한 배포 구조를 준비한다.
- SQLite baseline 스키마를 PostgreSQL 마이그레이션으로 전환한다.
- 앱 서버 인메모리 임시 캐시를 Redis 기반 캐시로 전환한다.
- 한국관광공사 OpenAPI 호출 계층을 추가한다.
- 유저 단위 30분 임시 캐시를 Redis에 구현한다.
- 위치, 카테고리, 선호도 기반 추천 점수화를 구현한다.
- 퀘스트 목록과 지도 화면의 목업 데이터를 실제 추천 결과로 교체한다.

### 11.3 3단계: 사용자 기록과 보상 시스템

- 네이버·구글 등 소셜 로그인 기반 회원가입·회원 식별 기능을 추가한다.
- 퀘스트 완료 기록, XP, 레벨, 뱃지 진행도를 저장한다.
- 모험가의 수첩을 실제 완료 이력 기반으로 표시한다.

### 11.4 4단계: Gemini와 리워드 확장

- Gemini 기반 퀘스트 후보 생성을 추가한다.
- AI 생성 결과 검증 정책을 구현한다.
- 퀘스트 완료, 뱃지 획득, 레벨 상승을 기준으로 `RewardOffer`와 `UserReward`를 발급하는 구조를 구현한다.

### 11.5 5단계: 지역 상권 파트너 연계

- 파트너 상점 등록과 승인 흐름을 구현한다.
- 파트너 담당자용 포털을 추가한다.
- 쿠폰 QR 코드 또는 사용 코드 검증 기능을 구현한다.
- `RewardRedemption` 기반 중복 사용 방지와 사용 내역 조회를 구현한다.
- 상점 로고, 쿠폰 이미지, 공유 카드 같은 정적 자산을 Object Storage와 CDN으로 제공한다.
- 지역 축제, 전통시장, 지역 상점과 연계한 기간 한정 리워드 정책을 운영할 수 있게 한다.

### 11.6 6단계: 고도화

- 연관 관광지·집중률(혼잡도) 기반 분산 추천을 추가한다. 이 기능은 실교통량·내비게이션 데이터 연계가 필요하다.
- 풀 PWA로 확장한다. 웹 푸시, 백그라운드 동기화, 오프라인 완료 큐잉을 추가한다.
- 사진 인증을 고도화한다. OCR 대조 범위를 넓히고 이미지 분류 기반 자동 판정을 도입한다.
- 사용자 원본 사진 공유를 콘텐츠 모더레이션과 함께 도입한다.

## 12. 검수 기준

### 12.1 문서 검수 기준

- `Design.md`의 필수 요구사항과 기능 항목이 모두 반영되어야 한다.
- 한국관광공사 OpenAPI 데이터 정책은 원본 데이터 영구 저장 금지와 유저 단위 임시 캐시 허용으로 일관되어야 한다.
- 현재 구현된 기능과 향후 구현할 기능이 분리되어야 한다.
- 문서는 한국어로 작성되어야 한다.
- API Key, `.env` 값, 개인 인증 정보가 포함되지 않아야 한다.
- OpenAPI 데이터 표시 화면과 공유 카드에 공공누리 출처표시가 포함되어야 한다.

### 12.2 기능 시나리오 검수 기준

- 사용자가 현재 위치를 기준으로 주변 관광지를 추천받는 흐름이 설명되어야 한다.
- 관광 카테고리 또는 지역이 변경되면 OpenAPI를 다시 호출하는 흐름이 설명되어야 한다.
- 퀘스트 완료 후 XP, 뱃지, 수첩 기록이 갱신되는 흐름이 설명되어야 한다.
- NAVER Maps REST API Key가 서버 프록시에서 보호되는 구조가 설명되어야 한다.
- 웹 서버와 앱 서버의 역할이 분리되어야 한다.
- 지역 상권 연계 시 파트너 포털, 리워드 발급, 쿠폰 사용 처리 흐름이 설명되어야 한다.
- 뱃지 단계 충족 시 꿈돌이 도감 항목이 해금되는 흐름이 설명되어야 한다.
- 사진 인증의 상호명 대조, EXIF 제거, 시스템 생성 공유 카드 흐름이 설명되어야 한다.
- 가입 시 위치정보·개인정보 수집·이용 동의와 만 14세 이상 연령 확인 흐름이 설명되어야 한다.

## 13. 설계상 결정 사항

- 새 설계 문서는 `PROJECT_DESIGN.md`로 관리한다.
- 기존 `Design.md`는 원본 요구사항 메모로 유지한다.
- 관광지 원본 데이터는 영구 저장하지 않는다.
- OpenAPI 응답은 유저 단위 임시 캐시에 한해 사용한다.
- 임시 캐시 기본 유효 시간은 30분으로 둔다.
- 관광 지역 또는 카테고리 변경 시 캐시를 폐기하고 OpenAPI를 다시 호출한다.
- 브라우저 캐시는 UX 보조 용도로 사용하고, 추천 및 퀘스트 생성 기준 데이터는 Redis 캐시에서 관리한다.
- 웹 서버와 앱 서버는 역할을 분리해 설계하고, 초기에는 각각 1대 규모로 시작할 수 있다.
- DB 엔진은 PostgreSQL로 확정한다.
- 서버 캐시는 Redis로 확정한다.
- 초기에는 하나의 관리형 PostgreSQL 데이터베이스를 사용하고, 사용자 데이터와 퀘스트 데이터는 schema 또는 table 구조로 논리 분리한다.
- 검증 완료된 퀘스트는 개인 과거 목록이 아니라 모든 사용자가 재사용할 수 있는 `ReusableQuest` 공용 자산으로 저장한다.
- 지역 상권 연계는 `PartnerStore`, `PartnerAccount`, `RewardOffer`, `UserReward`, `RewardRedemption`을 중심으로 확장한다.
- 퀘스트 생성은 템플릿 기반을 기본으로 하고, Gemini는 보조 수단으로 사용한다.
- 제출용 운영 규모는 전체 사용자 1,000명 이하, 동시 접속 50명 이하를 상한 기준으로 한다.
